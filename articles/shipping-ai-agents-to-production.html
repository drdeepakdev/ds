Shipping AI Agents to Production: What the Demos Don't Tell You

Anyone can wire an LLM to a few tools and record a jaw-dropping demo. Keeping that agent reliable, safe and affordable in front of real users is a different discipline entirely. Here's what I've learned building and reviewing agent systems this year.

DS
Deepu SomanPublished Jul 6, 2026 · 14 min read
in 𝕏

In the last eighteen months, "agent" went from a research term to the default architecture for AI products. The pattern is seductive: give a model a goal, a set of tools and a loop, and it appears to do the work of a junior engineer, analyst or support rep. The demos are real — I've built them. The problem is that a demo only has to succeed once, while production has to succeed on the ten-thousandth slightly-weird request, at 2 a.m., on someone else's data.

This article is the checklist I wish someone had handed me before my first agent went live. It's organised around the four places agent projects actually fail: reliability, safety, evaluation and cost.

1. Reliability: design for the loop going wrong

The core of every agent is a loop: observe → decide → act → observe. Every iteration is a chance for the model to drift, and drift compounds. The single biggest mindset shift is to stop treating the model as the system and start treating it as one unreliable component inside a system you control.

Bound everything

Every production agent needs hard, non-negotiable bounds enforced outside the model:

  • Step budget — a maximum number of tool calls per task. When it's exhausted, the agent must summarise progress and stop, not silently keep burning tokens.
  • Wall-clock budget — users forgive a 30-second wait with a progress indicator; they don't forgive a spinner that never resolves.
  • Blast radius — an allowlist of tools per task type. A refund-processing agent has no business calling the user-deletion API, even if a clever prompt asks it to.
# The harness owns the loop — the model never does
for step in range(MAX_STEPS):
    action = model.decide(context)
    if action.tool not in ALLOWED_TOOLS[task.kind]:
        return refuse("tool_not_permitted", action)
    result = execute_with_timeout(action, seconds=20)
    context.append(result)
    if action.is_final:
        break
else:
    return summarise_and_stop(context)  # budget exhausted

Make actions idempotent and reversible

Agents retry. Networks fail mid-call. If your create_invoice tool isn't idempotent, you will eventually double-bill a customer — the only question is when. Give every mutating tool an idempotency key, and prefer designs where the agent stages changes (a draft, a pending state) that a cheap deterministic step commits.

2. Safety: the prompt is not a security boundary

Everything that enters the context window — retrieved documents, web pages, tool output, user uploads — is input to the decision process. If an attacker can write text your agent will read, they can attempt to steer it. This is prompt injection, and no system prompt fully prevents it.

Treat your agent like a talented new hire with no security clearance: helpful, fast, and absolutely not allowed to move money on their own say-so.

Practical mitigations, in order of value:

  1. Capability scoping — the agent physically cannot call tools outside its task allowlist (see above). This is the one control that works even when persuasion succeeds.
  2. Human-in-the-loop gates — irreversible or high-value actions (payments, deletions, external emails) queue for one-click human approval. Design the approval UI to show what will happen, not the agent's reasoning, which can itself be manipulative.
  3. Content provenance — tag context segments by trust level (user input vs. retrieved third-party content) so at minimum you can audit which segment preceded a bad action.
  4. Output filtering — validate structured outputs against schemas; strip URLs and PII from free text where the channel doesn't need them.

Rule of thumb: ask "what's the worst thing this agent could do if every word in its context were written by an attacker?" If the answer scares you, the fix belongs in capabilities and gates — not in a sterner system prompt.

3. Evaluation: your test suite is now a dataset

Traditional tests assert exact outputs; agents don't have exact outputs. What replaces the test suite is an eval harness: a versioned dataset of tasks, an automatic grader, and a dashboard you check before every model, prompt or tool change.

Start embarrassingly small. Twenty real tasks with pass/fail criteria beat a thousand synthetic ones. Sources, in order of quality:

  • Real production transcripts where the agent failed (gold — add every one to the set).
  • Tasks your support team handles today, with their resolution as reference.
  • Synthetic edge cases generated by a stronger model, then human-reviewed.

Grade with the cheapest thing that works: exact-match on structured fields, then rubric-based LLM-as-judge for free text, then humans for a sampled slice. Track three numbers per release: task success rate, intervention rate (how often humans had to step in) and cost per solved task. If a prompt change moves success from 84% to 86% but doubles cost per task, you need the numbers to see it.

4. Cost: the loop multiplies your token bill

A chat completion costs what one call costs. An agent task costs steps × context size × price, and context grows every step. Three levers matter most:

LeverTypical savingHow
Context pruning30–60%Summarise or drop old tool results; keep only what the next decision needs.
Model tiering40–70%Route classification, extraction and formatting steps to a small model; reserve the frontier model for planning.
Prompt cachingup to 90% on repeated prefixesKeep the static system prompt and tool definitions in a cacheable prefix; put volatile data last.

Instrument cost per task from day one. The teams that get surprised by six-figure bills are always the ones measuring cost per request instead of per task.

The pre-launch checklist

  1. Step, time and tool budgets enforced by the harness, not the prompt.
  2. Every mutating tool idempotent; irreversible actions behind human approval.
  3. Eval set of ≥20 real tasks wired into CI; success, intervention and cost tracked per release.
  4. Prompt-injection tabletop: worst-case walkthrough with capabilities, not prompts, as the answer.
  5. Kill switch: one config flag that downgrades the agent to suggest-only mode.

None of this is glamorous, and all of it is the difference between a demo and a product. The teams shipping agents successfully in 2026 aren't the ones with secret prompting tricks — they're the ones doing systems engineering around a probabilistic core.
