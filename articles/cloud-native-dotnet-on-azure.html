Cloud-Native .NET on Azure: A 2026 Reference Stack

"We're a Microsoft shop" used to mean IIS, a monolith and a prayer. In 2026 it can mean one of the most productive cloud-native stacks available. This is the opinionated setup I recommend when a .NET team asks where to start.

DS
Deepu SomanPublished Apr 13, 2026 · 12 min read
in 𝕏

The .NET platform quietly became one of the best places to build cloud software: a fast modern runtime, first-class async, source-generated serialisation, and a cloud (Azure) that treats it as the favourite child. But the menu of Azure services is enormous, and most teams don't need a menu — they need a default order. Here's mine.

1. The reference stack at a glance

2. .NET Aspire: the dev-loop glue

Aspire is the piece that surprises people who left .NET years ago. It's an opinionated app-composition model: you declare your services and their dependencies in C#, and Aspire wires up local containers, connection strings, service discovery and a live dashboard with traces — no hand-maintained docker-compose, no "works on my machine" env files.

// AppHost/Program.cs — the whole local topology
var builder = DistributedApplication.CreateBuilder(args);

var db    = builder.AddPostgres("db").AddDatabase("orders");
var cache = builder.AddRedis("cache");
var bus   = builder.AddAzureServiceBus("bus");

builder.AddProject<Projects.OrdersApi>("api")
       .WithReference(db).WithReference(cache).WithReference(bus);

builder.Build().Run();

Hit F5 and the dashboard shows every service, its logs, and distributed traces across them. The same composition graph then informs deployment manifests. For onboarding alone — new starter productive in an hour — Aspire pays for itself.

3. Picking compute: Container Apps as the default

Azure Container Apps (ACA) is serverless containers on managed Kubernetes you never see: KEDA-based scale (including scale-to-zero), built-in HTTPS ingress, revisions for blue/green, and Dapr if you want it. It hits the sweet spot for 90% of line-of-business services: more control than Functions, none of AKS's cluster tax.

  • Azure Functions — keep for genuinely event-shaped glue (queue triggers, timers), not whole applications.
  • AKS — choose it when you have platform engineers who want Kubernetes: custom operators, service mesh, multi-tenant clusters.
  • App Service — still fine for a single web app; less compelling once you're container-first.

4. Identity and configuration: delete your connection strings

The single highest-leverage security move on Azure: managed identities everywhere. Your container app gets an identity from the platform; SQL, Service Bus, Key Vault and Storage all accept Entra ID tokens; and suddenly there are no secrets to rotate, leak or commit.

// no password anywhere in this program
builder.Configuration.AddAzureAppConfiguration(o =>
    o.Connect(new Uri(appCfgUrl), new DefaultAzureCredential())
     .ConfigureKeyVault(kv => kv.SetCredential(new DefaultAzureCredential())));

Locally, DefaultAzureCredential falls back to your developer login; in CI, to OIDC federated credentials from GitHub Actions. The same code, zero stored secrets, all the way to production.

5. Observability and delivery

.NET's OpenTelemetry support is now mature enough to be boring: add the OTel packages (or just Aspire's service defaults), and traces, metrics and logs flow to Azure Monitor with correlation intact. Wire the three golden alerts before launch: availability probe, p95 latency, and dead-letter queue depth.

For delivery, Bicep templates in the repo, GitHub Actions with environment approvals, and federated credentials instead of publish profiles. Deploy on merge to a staging revision, promote by shifting traffic — ACA revisions make canary releases a config value, not a project.

None of these choices is exotic, and that's the point: the 2026 Microsoft stack rewards boring-on-purpose. Pick the defaults, spend your innovation budget on your product, and let the platform absorb the undifferentiated heavy lifting it was built to absorb.
