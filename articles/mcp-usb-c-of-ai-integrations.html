MCP: The USB-C Moment for AI Integrations

Before USB-C, every device shipped its own cable. Before the Model Context Protocol, every AI app shipped its own bespoke integration for every tool. MCP collapsed that N×M mess into N+M — and if you build software, you'll want to speak it.

in 𝕏

Every generation of computing gets its interoperability moment. Databases got ODBC, the web got REST, and AI assistants got the Model Context Protocol — an open standard, introduced by Anthropic in late 2024 and since adopted across the industry, that defines how AI applications discover and call external tools, read data sources, and use prompt templates.

1. The N×M problem

Suppose you have M AI applications (a chat assistant, an IDE agent, an internal copilot) and N systems they should reach (GitHub, your CRM, a database, Slack). Without a standard, that's N×M custom integrations, each with its own auth, schemas and bugs. Every new assistant repeats all N integrations; every new tool must court every assistant.

MCP restructures this: each system exposes one MCP server; each AI app implements one MCP client. N+M pieces, and everything composes. That's the same economics that made USB-C inevitable — which is why adoption was so fast once the big model vendors and IDEs shipped clients.

2. How MCP works

MCP is JSON-RPC over a transport (stdio for local servers, streamable HTTP for remote ones). A server advertises three kinds of capability:

  • Tools — functions the model may call, each with a JSON Schema for arguments (search_orders, create_ticket).
  • Resources — readable data addressed by URI (file:///readme, crm://accounts/42), for context rather than action.
  • Prompts — reusable, parameterised prompt templates the client can surface to users.

The client (the AI app) lists capabilities at connect time, presents them to the model, executes the model's calls, and streams results back. Crucially, the host application stays in charge: it decides which servers to connect, which tools to expose, and when to ask the human for approval.

3. Build a server in an afternoon

The official SDKs make a minimal server almost boilerplate-free. Here's a TypeScript server exposing one tool:

import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({ name: "orders", version: "1.0.0" });

server.tool(
  "search_orders",
  "Search customer orders by email or order id",
  { query: z.string().describe("email or order id") },
  async ({ query }) => {
    const rows = await db.searchOrders(query);
    return { content: [{ type: "text", text: JSON.stringify(rows) }] };
  }
);

await server.connect(new StdioServerTransport());

Point any MCP-capable client at that binary and the model can now search orders — no custom plugin system, no per-assistant glue. The same server works in your IDE agent, your support copilot and your CLI.

Design advice from servers I've built and reviewed:

  • Fewer, richer tools. Ten overlapping CRUD tools confuse models; three task-shaped tools (find, summarise, act) perform better.
  • Descriptions are prompts. The tool description is the model's only manual — write it like documentation, with examples and constraints.
  • Return errors as data. A structured {"error": "not_found"} teaches the model to recover; a thrown exception teaches it nothing.

4. Security notes you cannot skip

An MCP server is an API gateway with a very persuadable client on the other end. Treat it that way:

  1. AuthN/AuthZ live in the server. Never trust the model to enforce who can see what. Remote servers should use OAuth and scope tokens per user.
  2. Beware tool poisoning. A malicious server's tool descriptions enter the model's context — vet third-party servers like you vet npm packages.
  3. Least privilege by default. Read-only tools separated from mutating ones, so hosts can expose safe subsets.

5. Where this goes

The interesting shift is organisational: teams now ship an MCP server alongside their REST API, the way they once shipped an SDK. Your service's "AI surface" is becoming a first-class product interface — designed, versioned and documented for a model audience as much as a human one. If you own a platform, the question is no longer whether to expose it to agents, but how well.
