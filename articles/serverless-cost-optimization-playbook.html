The Serverless Cost Optimisation Playbook

Serverless bills are made of millions of tiny decisions nobody reviewed. The good news: they're also fixed by a handful of big levers. Here are the nine I reach for, roughly in order of payoff.

DS
Deepu SomanPublished Apr 27, 2026 · 11 min read
in 𝕏

Serverless pricing feels like a rounding error — fractions of a cent per invocation — right up until the invoice arrives with eight digits after the currency symbol. The pattern is always the same: no single expensive thing, just staggering volume multiplied by small inefficiencies. Which means optimisation is also mechanical: find the multipliers, shrink them.

1. First, actually know the cost model

For function platforms (AWS Lambda, Azure Functions, Cloud Functions) you pay roughly: invocations × duration × memory, plus the services around them — API gateways, queues, storage operations, egress, and logging. Two implications people miss:

  • The bill is often not dominated by compute. Log ingestion and per-request gateway charges regularly exceed the function cost itself.
  • Duration is billed while you wait. A function idling 800ms on a downstream API pays for 800ms of memory-seconds doing nothing.

2. The nine levers

  1. Right-size memory. Memory and CPU scale together; more memory can finish faster and cost less. Run a power-tuning sweep per function instead of guessing. Typical finding: half your functions are over-provisioned, a few critical ones are under.
  2. Kill chatty fan-outs. One request that triggers 40 downstream invocations (function → queue → function → …) pays 40 sets of per-invocation overhead. Collapse internal hops; a function calling a library beats a function calling a function.
  3. Batch everything batchable. Process queue messages 25 at a time instead of 1. Same work, 1/25th of the invocation and polling overhead.
  4. Cache at the edge of your system. A 60-second cache on a hot read endpoint can remove millions of invocations a day. Users can't tell; your bill can.
  5. Tame your logs. Structured logs at INFO for the 0.1% of requests that matter (errors, samples), not debug dumps for all of them. Log ingestion is the most common "why is this line item huge?" answer.
  6. Shorten timeouts, fail fast. A 30s timeout on a function that should finish in 2s means bugs and retry storms bill you 15× worst-case.
  7. Move steady load off serverless. Serverless wins on spiky and low duty-cycle traffic. A function running near-continuously is a container's job at a fraction of the price. Do the crossover math per workload.
  8. Async by default. If the caller doesn't need the answer now, queue it — you smooth spikes, batch better, and stop paying for synchronous waiting.
  9. Delete zombie infrastructure. Scheduled functions for features that shipped two roadmaps ago, debug queues, orphaned storage triggers. Audit quarterly; the savings are embarrassing.

3. Architecture-level wins

The biggest saves come from moving work, not tuning it:

# Before: pay per read, forever
client → API gateway → function → database        # every single time

# After: pay once per change
change event → function → render to object storage/CDN
client → CDN                                       # ~free at scale

Precompute-on-write beats compute-on-read whenever reads dominate writes — which for content, dashboards and catalogs is almost always. The serverless part of your system shrinks to the write path, and the read path becomes static delivery.

4. Guardrails so it never surprises you again

  • Budgets with alerts at 50/80/100% of expected spend, per environment. A test loop invoking a function recursively should page someone before it pages the CFO.
  • Cost per business transaction as a tracked metric ("cost per order processed"), so growth and waste stop being conflated.
  • Concurrency limits on every function. They're your circuit breaker when a retry storm or infinite loop hits.
  • Tag everything by service and team, or the bill stays a mystery novel.

Serverless remains the right default for spiky, event-driven work — the economics are genuinely good. They just aren't unconditionally good. Treat cost as a first-class non-functional requirement, review it like you review performance, and the platform keeps its promise.
